    C-Model 样本自适应偏移 (SAO) 架构总结
    Sample Adaptive Offset Architecture & Implementation

    1. 全局流水线架构与数据流

    SAO 过程以 CTU (或 LCU) 为基本处理单元，发生在 Deblock (去块滤波) 之后。整体流水线如下：

    graph LR A["1. Deblock 像素输入
    (getPicYuvAddr_recon)"] --> B["2. RDO 模式决策
    (rdoSaoUnitAll)"] B --> C{"选择最佳模式"} C -- SAO_BO --> D["3. Band Offset 偏移应用"] C -- SAO_EO --> E["4. Edge Offset 偏移应用"] C -- SAO_OFF --> F["5. 原始像素跳过"] D --> G["6. 结果写回与上下文保存"] E --> G F --> G
    图 1：SAO CTU 级别流水线处理拓扑
    💡 核心机制：
    1. 上下文管理 (Context)：SAO 依赖于左侧和上方的边界像素，因此会在 save_sao_context 中把 CTU 的下边缘和右边缘缓存到 lineUp 和 colLeft 中。
    2. RDO (率失真优化)：在正式滤波前，通过 rdoSaoUnitAll 遍历所有可能的模式（BO、4种 EO 方向、Merge Left、Merge Up）以及不同的 offset 值，计算出代价 (RD Cost) 最小的配置。

    2. Band Offset (BO - 带偏移) 模式

    BO 模式不考虑像素的空间相关性，而是直接根据像素的亮度/色度强度值 (Pixel Value) 来分类，对特定强度区间的像素施加补偿。

    2.1 物理映射与原理

    BO 带划分 (以 8-bit 为例，0~255)：
    选中的连续 4 个 Band (子带)
    0 255
    图 2：BO 将 0-255 的像素区间均分为 32 个带，只对连续的 4 个带应用 offset
    架构实现细节：
    在 C-Model 的 processSaoUnitPic 中，通过提取像素的最高 5 bit (pixel >> 3) 作为 band_index。
    如果该 index 命中了 RDO 选出的起步 `subTypeIdx` 后连续 4 个带中的一个，就会将对应的 offset[i] 加到像素上。

    2.2 偏移量 (Offset) 精确计算公式与举例

    偏移量的确定分为“计算初始预估值”和“迭代寻优”两步 (参考源码 estIterOffset)：

    • 步骤一：估算均值 (硬件友好的除法近似)
      收集该类别内所有原始像素与重建像素的误差和 offsetOrg = ∑(Org - Rec)，以及像素个数 count。
      理想偏移应当是平均误差 offsetOrg / count，但在 C-Model 中为了避免除法：
      令 c = floor(log2(count))，初始预估值 init_guess = offsetOrg >> c。
      最后将 init_guess 截断至合法范围（例如 8-bit 下通常为 [-7, 7]）。
    • 步骤二：率失真迭代寻优
      候选 offset 从 init_guess 开始向 0 递减尝试。对于每个 offset：
      相对失真 ΔDist = (count * offset * offset - offsetOrg * offset * 2) >> shift
      预估码率 Rate = ABS(offset) + 常数
      计算 Cost = ΔDist + λ * Rate，选取 Cost 最小的 offset 作为最终输出。
    📝 详细举例计算：
    假设在一个 CTU 中，某个 BO Band 内有 8 个像素 (count = 8)。
    它们经过 Deblock 后的 (原始值 - 重建值) 误差分别为：+4, +5, +3, +4, +5, +3, +4, +4。
    • offsetOrg = 32。
    • c = floor(log2(8)) = 3。
    • init_guess = 32 >> 3 = 4。所以从 4 开始试探。
    假设进入迭代寻优：
    尝试 offset = 4：ΔDist = 8 * 4² - 32 * 4 * 2 = 128 - 256 = -128。需要传输的码率 Rate ≈ 4 位，Cost↔4 = -128 + 4 * λ。
    尝试 offset = 3：ΔDist = 8 * 3² - 32 * 3 * 2 = 72 - 192 = -120。需要传输的码率 Rate ≈ 3 位，Cost↔3 = -120 + 3 * λ。
    如果 λ 较小（即看重画质），算法会最终确定偏移量为 4；如果 λ 很大（码率吃紧，大于 8），则可能退化选择 3。代码中包含一个 while (iterOffset != 0) 的循环，会按照 4、3、2、1 的顺序一直尝试到 0 为止，最终选取整个过程中 Cost 最小的那一个。

    2.3 连续 4 个起步带 (subTypeIdx) 是如何确定的？(滑动窗口寻优)

    虽然像素全域被划分为了 32 个 Band，但考虑到编码开销，SAO 规定每个 CTU 最多只能选取连续的 4 个 Band 进行补偿。这 4 个连续 Band 的起步位置 (subTypeIdx) 是通过滑动窗口扫描决定的：

    在 C-Model 的代码中，这是一个由 i (从 0 遍历到 28) 控制的 for 循环。对于每个起步位置 i，编码器会将 i, i+1, i+2, i+3 这 4 个带独立算出来的最优代价 tempCost 加起来，最后挑选出总代价最小的那一组，将其起步位置记作 subTypeIdx 写入码流。

    3. Edge Offset (EO - 边缘偏移) 模式

    EO 模式利用像素间的局部空间梯度，比较当前像素与其邻居的大小关系，根据分类 (Class) 来补偿。

    3.1 EO 的四种方向

    graph TD A((当前像素 C)) B1[EO_0: 水平] --> A B2[EO_1: 垂直] --> A B3[EO_2: 135度] --> A B4[EO_3: 45度] --> A

    3.2 边缘分类 (Edge Classes)

    根据与前后两个像素（p_left, p_right）的大小比较，将边缘分为 4 种有效类和 1 个跳过类：

    Class 1 (局部极小)
    C < L 且 C < R
    + 正偏移
    Class 2 (谷底边缘)
    C < L, C == R (或反之)
    + 正偏移
    Class 3 (山顶边缘)
    C > L, C == R (或反之)
    - 负偏移
    Class 4 (局部极大)
    C > L 且 C > R
    - 负偏移
    💡 设计约束背后深层的物理意义 (防振铃效应)：
    为什么 C-Model 强硬规定 Class 1/2 必须是正偏移，Class 3/4 必须是负偏移？
    这是因为在视频压缩中，高频量化会引发著名的 Gibbs现象 (振铃效应)：
    1. 谷底 (Class 1) 经常发生 Undershoot (下冲)：原本是 10 的亮度，重建后变成了 5 (变得更暗)。所以 Org - Rec > 0，必须加正偏移把它拉亮回来。
    2. 山顶 (Class 4) 经常发生 Overshoot (过冲)：原本 200 的亮度，重建后飙到了 210 (变得过曝)。所以 Org - Rec < 0，必须加负偏移把它压暗回来。
    如果在谷底算出了负偏移，说明数据是异常的噪点而非典型的量化失真，C-Model 会直接将 offset 强制归 0，放弃补偿。

    3.3 EO 偏移量精确计算与举例

    EO 偏移量的计算在 estIterOffset 中受到严格的符号约束，其数值通常很小，在 8-bit 视频中最大偏移除外被限制在 [-7, 7] 之间。

    📝 EO 精确举例计算：
    假设在一个 CTU 中使用 水平方向 (EO_0) 进行扫描，找出了 16 个属于 Class 1 (局部极小/谷底) 的像素 (count = 16)。
    因为量化的下冲效应，这 16 个像素普遍比原图暗，它们的 (原始值 - 重建值) 误差和为：offsetOrg = +48。
    • 估算均值： c = floor(log2(16)) = 4。
    • 初始预估值： init_guess = 48 >> 4 = 3。因为是正数，符合 Class 1 的要求，保留。
    假设进入迭代寻优：
    尝试 offset = 3：ΔDist = 16 * 3² - 48 * 3 * 2 = 144 - 288 = -144。码率 Rate = ABS(3) + 1 = 4，Cost↔3 = -144 + 4 * λ。
    尝试 offset = 2：ΔDist = 16 * 2² - 48 * 2 * 2 = 64 - 192 = -128。码率 Rate = ABS(2) + 1 = 3，Cost↔2 = -128 + 3 * λ。
    最终通过比较 Cost，编码器会决定给这 16 个谷底像素统一加上 +3 或 +2 的亮度补偿，以此抹平振铃下冲。
    • 在 estIterOffset 寻优时，因为只有单向尝试（非正或非负），因此迭代只会朝一个确定方向归零。
    • 计算位元消耗 tempRate 时，EO 的开销模型为 ABS(offset) + 1 (除了边界修正)，比 BO 略少，因为 EO 绝对不需要传输偏移符号。

    4. 率失真优化 (RDO) 最佳模式抉择

    在 rdoSaoUnitAll 函数中，编码器会遍历所有可能的模式，并最终抉择出使得 RD Cost 最小的模式。其抉择逻辑如下：

    CTU 级打擂台流程 (C-Model 实现)

    1. 前置基础数据 (calcSaoStatsCuOrg)：CTU 在进入宏观抉择前，已经利用前文提到的微观迭代算法，独立求出了各个 BO 频带和 4 种 EO 方向各 Class 的最佳 offset 及其对应的最小代价 tempCost。
    2. 汇总各模式的总代价：
      • BO 的总代价：尝试 32 种可能的起步带 (subTypeIdx)，将连续 4 个带的最佳微观 tempCost 累加，选出总代价最小的起步带组合。
      • EO 的总代价：分别针对水平、垂直、135度、45度四种方向，将该方向下 4 个 Class 的微观 tempCost 累加，得出四种方向各自的总代价。
    3. 考察相邻 Merge 模式：额外尝试 Merge Left (直接复用左侧 CTU 算好的所有参数) 和 Merge Up (复用上方 CTU 参数)。Merge 模式因为不需要写入具体的 offset 和模式参数，其宏观 Rate 极低，总 Cost 往往极具竞争力。
    4. 终极抉择：比较 SAO_OFF (不滤波)、SAO_BO (最佳连续4带组合)、SAO_EO (4个方向中最佳的那个)、Merge Left 和 Merge Up 的总代价，选择 Cost 最小的配置，作为整个 CTU 的终极模式并写入 saoLcuParam。
