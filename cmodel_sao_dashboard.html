    C-Model 样本自适应偏移 (SAO) 架构总结
    Sample Adaptive Offset Architecture & Implementation

    1. 全局流水线架构与数据流

    SAO 过程以 CTU (或 LCU) 为基本处理单元，发生在 Deblock (去块滤波) 之后。整体流水线如下：

    graph LR A["1. Deblock 像素输入
    (getPicYuvAddr_recon)"] --> B["2. RDO 模式决策
    (rdoSaoUnitAll)"] B --> C{"选择最佳模式"} C -- SAO_BO --> D["3. Band Offset 偏移应用"] C -- SAO_EO --> E["4. Edge Offset 偏移应用"] C -- SAO_OFF --> F["5. 原始像素跳过"] D --> G["6. 结果写回与上下文保存"] E --> G F --> G
    图 1：SAO CTU 级别流水线处理拓扑
    💡 核心机制：
    1. 上下文管理 (Context)：SAO 依赖于左侧和上方的边界像素，因此会在 save_sao_context 中把 CTU 的下边缘和右边缘缓存到 lineUp 和 colLeft 中。
    2. RDO (率失真优化)：在正式滤波前，通过 rdoSaoUnitAll 遍历所有可能的模式（BO、4种 EO 方向、Merge Left、Merge Up）以及不同的 offset 值，计算出代价 (RD Cost) 最小的配置。

    2. Band Offset (BO - 带偏移) 模式

    BO 模式不考虑像素的空间相关性，而是直接根据像素的亮度/色度强度值 (Pixel Value) 来分类，对特定强度区间的像素施加补偿。

    2.1 物理映射与原理

    BO 带划分 (以 8-bit 为例，0~255)：
    选中的连续 4 个 Band (子带)
    0 255
    图 2：BO 将 0-255 的像素区间均分为 32 个带，只对连续的 4 个带应用 offset
    架构实现细节：
    在 C-Model 的 processSaoUnitPic 中，通过提取像素的最高 5 bit (pixel >> 3) 作为 band_index。
    如果该 index 命中了 RDO 选出的起步 `subTypeIdx` 后连续 4 个带中的一个，就会将对应的 offset[i] 加到像素上。

    2.2 偏移量 (Offset) 计算策略

    偏移量的计算不是简单求均值，而是通过 estIterOffset 结合率失真代价进行迭代寻优的：

    • 初始统计：遍历 CTU，累加当前频带下原始像素与重建像素的差值和 offsetOrg = ∑(Org - Rec)，以及属于该频带的像素个数 count。
    • 迭代搜索：令候选偏移量 offset 从初始均值开始，逐步向 0 递减。
    • 失真计算公式：C-Model 中的 estSaoDist 公式为：
      ΔDistortion = (count * offset * offset - offsetOrg * offset * 2) >> shift
      (这是通过展开平方差公式化简得到的相对失真变化量)

    3. Edge Offset (EO - 边缘偏移) 模式

    EO 模式利用像素间的局部空间梯度，比较当前像素与其邻居的大小关系，根据分类 (Class) 来补偿。

    3.1 EO 的四种方向

    graph TD A((当前像素 C)) B1[EO_0: 水平] --> A B2[EO_1: 垂直] --> A B3[EO_2: 135度] --> A B4[EO_3: 45度] --> A

    3.2 边缘分类 (Edge Classes)

    根据与前后两个像素（p_left, p_right）的大小比较，将边缘分为 4 种有效类和 1 个跳过类：

    Class 1 (局部极小)
    C < L 且 C < R
    + 正偏移
    Class 2 (谷底边缘)
    C < L, C == R (或反之)
    + 正偏移
    Class 3 (山顶边缘)
    C > L, C == R (或反之)
    - 负偏移
    Class 4 (局部极大)
    C > L 且 C > R
    - 负偏移
    💡 设计约束：
    Class 1 和 2 只能使用正偏移 (提亮谷底)，Class 3 和 4 只能使用负偏移 (压暗山顶)。这在 C-Model 的码流解析 (codeSaoOffset) 和推导逻辑中是严格约束的。

    3.3 EO 偏移量计算与抉择

    与 BO 类似，EO 的偏移量计算同样统计各类别的 offsetOrg 和 count。但在 C-Model 代码中，EO 受到严格的符号约束：

    • 在 estIterOffset 寻优时，若为 Class 1/2，候选 offset 强制非负；若为 Class 3/4，候选 offset 强制非正。
    • 计算位元消耗 tempRate 时，EO 的开销模型为 ABS(offset) + 1 (除了边界修正)，比 BO 略少，因为 EO 不用传输偏移符号。

    4. 率失真优化 (RDO) 最佳模式抉择

    在 rdoSaoUnitAll 函数中，编码器会遍历所有可能的模式，并最终抉择出使得 RD Cost 最小的模式。其抉择逻辑如下：

    RDO 抉择流程 (C-Model 实现)

    1. 统计基础数据 (calcSaoStatsCuOrg)：先遍历 CTU，收集 32 个 BO 频带和 4 种 EO 方向各 4 个 Class 的 offsetOrg 和像素 count。
    2. 计算各模式的最优代价：
      • 对于 BO：尝试 32 种可能的起步带 (subTypeIdx)，将连续 4 个带的最佳 tempCost 累加。
      • 对于 EO：分别计算水平、垂直、135度、45度四种模式下，各 Class 的 tempCost 之和。
      • 代价公式 (Fix-point 实现)：tempCost = tempDist + ((lambda * tempRate) >> 16)
    3. 考察相邻 Merge 模式：尝试 Merge Left (拷贝左侧 CTU 参数) 和 Merge Up (拷贝上方 CTU 参数)。由于 Merge 模式无需传输大量参数，其 tempRate 极低，往往能获得更小的 Cost。
    4. 最终抉择：比较 SAO_OFF (不滤波)、SAO_BO (最佳起步带)、SAO_EO (最佳方向)、Merge Left 和 Merge Up 的总代价，选择 Cost 最小的配置，将结果写入 saoLcuParam 并进入后续 processSaoUnitPic 进行物理补偿。
