    HEVC EMC (Entropy Multi-Context) 硬件架构级解析
    CABAC 引擎与树形状态机底层揭秘

    1. EMC 模块定位与异步 FIFO 架构

    在 HEVC 的硬件流水线中，EMC（熵编码）模块位于全链路的最末端。它的核心职责是接收上游（如模式决策、变换量化、SAO等）产生的海量语法元素参数，并通过强大的 CABAC (Context-Adaptive Binary Arithmetic Coding) 引擎将它们极致压缩为最终的二进制码流 (Bitstream)。

    1.1 为什么需要庞大的 FIFO 阵列？

    ⚠️ 核心硬件时序痛点 (吞吐量不对等)：

    前端模块（如宏块级的 SAD/SATD 粗选、变换量化 T/Q）属于高度并行的 像素级并行处理，吞吐量极大。而 CABAC 算术编码是一个基于概率模型的 串行状态机，它必须根据前一个 Bin 编码后的状态来更新概率模型（Context Update），这使得它无法通过增加硬件面积来简单并行化。

    为了抹平这种恐怖的吞吐量鸿沟，硬件架构在 EMC 前端部署了庞大的 FIFO (First-In-First-Out) 队列阵列，用来平抑前端的突发数据峰值，实现“宽进严出”的时序解耦。

    根据 H2(HEVC)_emc_Architecture.docx 的规范，EMC 依赖于多个专用的 FIFO 来接收不同类型的数据：

    • SAO Control FIFO: 包含 SAO 的 Type 和 Offset。
    • Coding Unit Control FIFO: 包含宏块划分类型、预测模式 (Intra/Inter) 等。
    • CBF Flags FIFO: Coded Block Flags，表示该子块是否有非零残差。
    • MVD FIFO: 运动矢量差值。
    • Level Residual FIFO: 量化后的残差系数。

    2. 空间树形 FSM 状态机 (Quad-Tree Building)

    将一棵可变深度的 64x64 编码树 (Coding Tree) 转化为顺序流，是一个极其复杂的递归遍历过程（Z-Scan）。在高级软件（如 HM）中，这可以通过简单的函数递归（encodeCU 调用自身）来优雅实现。但在纯粹的硬件或 C-Model 中，硬件不支持函数递归，必须使用硬件寄存器阵列来模拟堆栈！

    2.1 基于 cb_cnt 数组的“虚拟堆栈”遍历法

    💡 数组追踪物理深度：
    在代码中，有一个核心数组 cb_cnt[4]。由于最大深度是 3 (64x64 切分为 32x32, 16x16, 8x8)，这个数组分别记录了在每个层级，当前正在处理的是第几个子节点（0, 1, 2, 或 3）。
    • cb_cnt[1]: 当前在 32x32 层的游标 (0~3)
    • cb_cnt[2]: 当前在 16x16 层的游标 (0~3)
    • cb_cnt[3]: 当前在 8x8 层的游标 (0~3)
    当底层游标满 4 时，自动清零并向上层游标进位，完美用纯状态机实现了复杂的 Z-Scan 树遍历！

    2.2 到达叶子节点时，FSM 在算什么？(Luma & Chroma 编码流)

    当 Z-Scan 状态机确认当前宏块 split_flag == 0（即到达叶子节点 CU）时，它不再向下进栈，而是**触发 CABAC 引擎**，对该 CU 的所有语法元素进行算术编码。

    当前 CU 节点的编码载荷 (CABAC Encoding Payload)
    • ① 预测模式信息 (Prediction Info):
      从 Coding Unit Control FIFO 中读取。首先编码 cu_skip_flag。如果未 skip，则编码预测模式 (Intra/Inter)。 对于 Inter 模式，编码 MVD (运动矢量差分) 和参考帧索引；对于 Intra 模式，编码预测方向。
    • ② 亮度与色度的残差处理 (Luma & Chroma Residuals):
      如果 root_cbf == 1，则进入 TU (Transform Unit) 变换树。HEVC 中，YUV 通道的残差系数是**严格按顺序**从 Level Residual FIFO 读取并编码的。
      严格的时序处理顺序: Luma (Y) ➔ Chroma Cb (U) ➔ Chroma Cr (V)
      4:2:0 降采样映射关系: 当前叶子 CU 的亮度块大小为 --，由于 4:2:0 色度抽样格式，它将关联 -- 大小的 Cb 块和 Cr 块。
      CABAC 会依次为这些块编码 cbf_luma, cbf_cb, cbf_cr。当对应的 cbf=1 时，才会将非零的残差系数量化值写入码流。

    2.3 CABAC 语法元素解析状态机 (CABAC Syntax Parsing FSM)

    状态机核算内容详解

    1. ST_CU_SPLIT
    计算目标： 当前宏块是否还要继续四叉树切分。
    结果用途： 指导空间树形 Z-Scan 状态机（上一节）是否执行进栈 (Depth++) 操作。
    2. ST_SKIP
    计算目标： 当前宏块是否是 Skip 模式（没有任何残差，且运动矢量直接 Copy 相邻块）。
    结果用途： 属于“一票否决”状态。如果是 Skip，则直接跳过后续所有的 PU/TU 计算，只算一个 Merge Index 就结束。这是实现极高压缩率的核心。
    3. ST_PRED_MODE
    计算目标： 是帧内预测 (Intra) 还是帧间预测 (Inter)。
    结果用途： 决定状态机进入左侧分支（处理空间纹理方向）还是右侧分支（处理时间运动矢量）。
    4. PU_INFO 层级
    计算目标： 预测阶段的具体参数。对于 Intra，算 35 种预测方向；对于 Inter，算 MVD (Motion Vector Difference) 以及参考帧索引。
    结果用途： 解码器依靠这些信息来生成“预测图像”。
    5/6. ROOT_CBF / TU_TREE
    计算目标： 预测图和原图做差得到的“残差”是否为全 0 (root_cbf)。如果不是全 0，进入 TU 变换树。
    结果用途： 避免对全 0 的大块残差进行无效扫描。
    7/8. CBF / COEFF
    计算目标： 从 Level Residual FIFO 依次读取 Luma、Cb、Cr 的量化系数。对于非零块，计算 Significant Map（哪些位置有非零系数）、绝对值 Abs 和符号 Sign。
    结果用途： 这是 HEVC 中占据码流最大头的部分，CABAC 引擎会对非零系数的位置进行极致的概率上下文压缩。

    3. 核心机制：CABAC 编码全链路剖析 (从 CU 语法元素到 Bitstream)

    很多初学者在学习 CABAC 时，往往会把“语法元素”和“底层算术引擎”割裂开来。实际上，它们是一条流水线。
    上一节的状态机产生了诸如 MVD、Skip_flag 等高层语法元素，这些元素必须经过二值化 (Binarizer) 变成纯粹的 0 和 1 (Bin)，才能送入底层的 算术引擎 (Arithmetic Engine) 进行压缩。

    🛠️ 深度解析：Binarizer (二值化器) 的三大核心兵器

    在把十进制的语法元素喂给底层算盘前，硬件的 Binarizer 模块必须按特定算法将其“切碎”成 0 和 1 (Bins)。因为算术编码本质上是在二叉树上寻路。HEVC 中主要使用以下三种二值化算法：

    1. 一元码 (Unary)

    最暴力的算法。数字 N 就用 N 个 1 加上一个 0 结尾来表示。

    举例：N = 3
    输出 Bins：1 1 1 0

    用途：小数值或受限数值，如 merge_idx、part_mode 等。

    2. 截断莱斯编码 (Truncated Rice, TR)

    将数字 X 拆分为商(q)和余数(r)。商用一元码编码，余数用定长二进制编码。参数 K 决定除数 (2^K)。

    举例：X=5, 参数K=1
    除数是 2^1=2。
    商 q = 5>>1 = 2 (一元码编为 1 1 0)
    余 r = 5&1 = 1 (长度为K=1，编为定长 1)
    输出 Bins：1 1 0 1

    用途：极高频出现，但数值范围较大的量化系数绝对值的剩余部分 (coeff_abs_level_remaining)。

    3. k阶指数哥伦布 (Exp-Golomb, EGk)

    专为大数值设计的动态长度编码。将数值加偏置后，拆分成“前缀”(表示长度的一元码)和“后缀”(纯二进制)。

    举例：X=1, EG0阶
    加偏置 X'=1+1=2 (二进制为 10)。
    拿掉最高位1，剩下 0 作为后缀。后缀长1位，所以前缀是1个0。
    标准输出是 0 1 0 (HEVC中习惯0和1翻转，所以输出 1 0)。

    用途：运动矢量差分 (MVD) 等跨度极大的数值。

    💡 硬件小科普：Range 是怎么确定的？
    在真实的 HEVC 硬件中，算术引擎包含两个核心寄存器：
    1. Range (区间)：9-bit 寄存器，在每个 Slice 开始时会固定初始化为 510 (0x01FE)，它代表当前可用的概率空间总长度。每次编码都会把这个长度按概率切小。
    2. Low (底线)：10-bit 寄存器，初始值为 0，代表当前编码状态在数轴上的物理起点。
