    1. EMC 模块定位与异步 FIFO 架构

    在 HEVC 的硬件流水线中，EMC（熵编码）模块位于全链路的最末端。它的核心职责是接收上游（如模式决策、变换量化、SAO等）产生的海量语法元素参数，并通过强大的 CABAC (Context-Adaptive Binary Arithmetic Coding) 引擎将它们极致压缩为最终的二进制码流 (Bitstream)。

    1.1 为什么需要庞大的 FIFO 阵列？

    ⚠️ 核心硬件时序痛点 (吞吐量不对等)：

    前端模块（如宏块级的 SAD/SATD 粗选、变换量化 T/Q）属于高度并行的 像素级并行处理，吞吐量极大。而 CABAC 算术编码是一个基于概率模型的 串行状态机，它必须根据前一个 Bin 编码后的状态来更新概率模型（Context Update），这使得它无法通过增加硬件面积来简单并行化。

    为了抹平这种恐怖的吞吐量鸿沟，硬件架构在 EMC 前端部署了庞大的 FIFO (First-In-First-Out) 队列阵列，用来平抑前端的突发数据峰值，实现“宽进严出”的时序解耦。

    根据 H2(HEVC)_emc_Architecture.docx 的规范，EMC 依赖于多个专用的 FIFO 来接收不同类型的数据：

    • SAO Control FIFO: 包含 SAO 的 Type 和 Offset。
    • Coding Unit Control FIFO: 包含宏块划分类型、预测模式 (Intra/Inter) 等。
    • CBF Flags FIFO: Coded Block Flags，表示该子块是否有非零残差。
    • MVD FIFO: 运动矢量差值。
    • Level Residual FIFO: 量化后的残差系数。

    2. 空间树形 FSM 状态机 (Quad-Tree Building)

    将一棵可变深度的 64x64 编码树 (Coding Tree) 转化为顺序流，是一个极其复杂的递归遍历过程（Z-Scan）。在高级软件（如 HM）中，这可以通过简单的函数递归（encodeCU 调用自身）来优雅实现。但在纯粹的硬件或 C-Model 中，硬件不支持函数递归，必须使用硬件寄存器阵列来模拟堆栈！

    2.1 基于 cb_cnt 数组的“虚拟堆栈”遍历法

    💡 数组追踪物理深度：
    在代码中，有一个核心数组 cb_cnt[4]。由于最大深度是 3 (64x64 切分为 32x32, 16x16, 8x8)，这个数组分别记录了在每个层级，当前正在处理的是第几个子节点（0, 1, 2, 或 3）。
    • cb_cnt[1]: 当前在 32x32 层的游标 (0~3)
    • cb_cnt[2]: 当前在 16x16 层的游标 (0~3)
    • cb_cnt[3]: 当前在 8x8 层的游标 (0~3)
    当底层游标满 4 时，自动清零并向上层游标进位，完美用纯状态机实现了复杂的 Z-Scan 树遍历！

    3. CABAC 数学归一化引擎 (Renormalization)

    CABAC 的魔法在于算术编码，它维护着两个核心物理寄存器：Range (当前概率区间的宽度) 和 Low (区间的起点)。当概率区间随着编码不断被切分而缩小，当 Range 小于一个阈值（0x100 即 256）时，硬件就必须执行归一化 (Renormalization) 动作，也就是把编码结果“挤”到码流里去。

    Range 寄存器 (9-bit)

    0x01FE
    (十进制: 510)
    ⚠️ Range < 0x0100，必须执行归一化！

    输出 Bitstream

    每次归一化左移，都会挤出 1 bit 写入码流 FIFO。
